## 1) Architecture/layout consistency
- Project layout is coherent: `AppShell` + module top bar + page content wrapper keep structure consistent.
- Main issue: project-change resets in planning pages are done imperatively during render via `if (prevRef.current !== singleProjectId) { setState(...) }`.
- Recommendation: use project-scoped derived state (or keyed remount) to avoid both render-phase and effect-phase synchronous state resets.

## 2) Component composition and reuse
- Good: shared primitives (`Button`, `Dialog`, `Badge`) and reusable planning components (`StoryDetailDialog`, `StoryForm`, `BacklogRow`).
- Gap: duplicated nav item rendering between desktop and mobile navigation.
- Recommendation: extract shared nav-link renderer to reduce drift risk (deferred).

## 3) Styling consistency (Tailwind/shadcn)
- Good: consistent tokens (`bg-card`, `text-muted-foreground`, border opacity patterns).
- Gap: mixed focus patterns (`focus-ring` utility vs bespoke `focus-visible:ring-*` chains).
- Recommendation: standardize on `focus-ring` for custom controls where practical (deferred except in touched files).

## 4) State/data-fetching patterns (RSC vs client)
- Good: dashboard page loads initial data server-side (`force-dynamic`) and hydrates client sections.
- Gap 1: `useAutoRefresh` interface implies periodic refresh but performs URL-change fetch only.
- Gap 2: some fetch flows lack abort handling.
- Recommendation: align hook behavior with API contract and add cancellation/visibility-aware polling.

## 5) Accessibility and interaction quality
- Good: many controls include `aria-label`, `aria-current`, and visible focus treatment.
- Gap 1: request row expansion control is not an actual keyboard-focusable control.
- Gap 2: mobile nav drawer is missing focus trapping.
- Recommendation: use explicit buttons for row expansion; add focus trap for modal nav (trap deferred).

## 6) Performance quick wins
- Prevent render-phase state updates in planning pages.
- Avoid stale updates by using cancellable refresh logic.
- Keep expensive rerenders bounded (existing memo usage is generally solid in planning board/list models).
